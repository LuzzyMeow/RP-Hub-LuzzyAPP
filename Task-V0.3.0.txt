展示格式：
↑ {输入tokens} (缓存 {缓存tokens} · {命中率}%) ↓ {输出tokens} · {tok/s} tok/s · {响应时间}s
**示例：**
↑ 2006 (缓存 1920 · 95.7%) ↓ 300 · 45.2 tok/s · 6.6s
**命中率为0时：**
↑ 2006 ↓ 300 · 45.2 tok/s · 6.6s
流式生成中的实时更新，生成过程中，统计行需实时滚动更新，分为三个阶段：
| 阶段 | 展示内容 | 触发时机 |
|------|---------|---------|
| **生成中** | `↓ {当前已输出tokens} · {当前tok/s} tok/s · {已用时间}s` | 每收到一个流式chunk更新一次 |
| **即将完成** | 同上，tok/s 和时间持续刷新 | 直到收到末token |
| **生成完成** | 完整格式：`↑ {输入} (缓存 · %) ↓ {输出} · {tok/s} tok/s · {总时间}s` | 拿到完整usage后一次性补全 |

**生成中示例：**
↓ 128 · 42.1 tok/s · 3.0s
**生成完成后示例：**
↑ 2006 (缓存 1920 · 95.7%) ↓ 300 · 45.2 tok/s · 6.6s
**实时计算逻辑：**
- `当前已输出tokens`：流式过程中逐chunk累计（OpenAI/Anthropic的chunk可能不带token计数，可用 `已输出字符数 / 估算系数` 近似，或等chunk携带delta usage时取精确值）
- `当前tok/s`：`当前已输出tokens / 已用时间(秒)`，每chunk刷新
- `已用时间`：从请求发出到当前时刻的秒数，每chunk刷新
- 生成完成后用精确的 usage 字段替换估算值，并补上输入tokens、缓存信息

**各提供商流式 usage 获取：**
| 提供商 | 流式中的usage位置 | 说明 |
|--------|------------------|------|
| **OpenAI** | 最后一个chunk（`[DONE]`之前） | 需设 `stream_options: {"include_usage": true}`，中间chunk无usage，需用近似估算 |
| **Anthropic** | `message_start`事件给输入tokens，`message_delta`事件给输出tokens | 输入tokens可提前获取，输出tokens在结束时获取 |
| **Gemini** | `usageMetadata`在流中返回 | URL加 `?alt=sse` 走SSE格式 |

数据来源（生成完成后精确值）
| 指标 | OpenAI | Anthropic | Gemini |
|------|--------|-----------|--------|
| 输入tokens | `usage.prompt_tokens` | `usage.input_tokens + cache_read + cache_write` | `usageMetadata.promptTokenCount` |
| 缓存tokens | `usage.prompt_tokens_details.cached_tokens` | `usage.cache_read_input_tokens` | 不适用 |
| 输出tokens | `usage.completion_tokens` | `usage.output_tokens` | `usageMetadata.candidatesTokenCount` |
| tok/s | `输出tokens / 总响应时间(秒)` | 同左 | 同左 |
| 响应时间 | 从请求发出到收到末token的实际秒数 | 同左 | 同左 |

**缓存命中率计算：**
- OpenAI：`cached_tokens / prompt_tokens × 100`
- Anthropic：`cache_read / (input_tokens + cache_read + cache_write) × 100`
- Gemini：不显示

二、重大升级

针对记忆系统的全局记忆功能，运行在 APP 内，现有实现方式为：
> 维护一个纯文本文件（.md）作为全局记忆，每次交互前全量注入，APP 内有预览框展示纯文本全部内容，用户可查看和编辑。

现需将其升级为 **ACE（Agentic Context Engineering）记忆机制**——从"无结构文本追加"升级为"结构化策略手册 + 反思学习闭环"，同时保留用户可查看和编辑的记忆预览能力。

ACE 核心设计速览

| 概念 | 说明 |
|------|------|
| **Skillbook** | 替代现有文本文件的结构化记忆对象，持久化为 JSON + 向量缓存两个文件 |
| **Skill（策略）（此Skill非技能SKILL）** | Skillbook 中的单条记忆，带元数据（id、分类、正文、有效性计数、启用状态） |
| **三步循环** | 每次交互后：执行 → 反思 → 更新策略手册 |
| **评分淘汰** | 每条策略根据使用反馈累计 helpful/harmful 计数，低分自动停用 |
| **去重** | 周期性用嵌入模型计算相似度，合并重复策略（可选） |

现有架构 → 目标架构映射

| 维度 | 现有 | 目标（ACE） |
|------|------|------------|
| 存储格式 | 纯文本文件 | `memories.json`（结构化数据）+ `memories.embeddings.npz`（向量缓存） |
| 记忆结构 | 无结构文本 | 每条策略含 id / category / content / helpful_count / harmful_count / active 等字段 |
| 写入方式 | 追加文本 | 三步循环：Agent 执行 → Reflector 反思 → SkillManager 增量更新（ADD/UPDATE/REMOVE/TAG） |
| 读取方式 | 全量读取注入 | 全量渲染 active 策略为 markdown 注入提示词 |
| 质量管理 | 无 | 评分淘汰 + 嵌入去重 |
| 学习来源 | 手动/追加 | 从 Agent 自身执行轨迹中提炼经验，无需外部标注 |
| 预览编辑 | 纯文本预览框 | 卡片列表 + 点击编辑（详见第六章） |

Skill 数据结构

```json
{
  "id": "mem-00001",
  "category": "user_preference",
  "content": "用户偏好用中文回复，颜文字多输出",
  "helpful_count": 5,
  "harmful_count": 0,
  "neutral_count": 1,
  "active": true,
  "created_at": "2026-06-20T18:30:00Z",
  "updated_at": "2026-06-20T19:00:00Z"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 唯一标识，格式 `mem-XXXXX`，自增 |
| category | string | 分类标签（如 user_preference / task_pattern / tool_usage） |
| content | string | 策略正文，描述一条经验或偏好 |
| helpful_count | int | 被评估为"有帮助"的次数 |
| harmful_count | int | 被评估为"有害"的次数 |
| neutral_count | int | 被评估为"中性"的次数 |
| active | bool | 是否启用，false 时停止注入但仍保留数据（软删除） |
| created_at | string | 创建时间，ISO 8601 |
| updated_at | string | 最后更新时间，ISO 8601 |

三步工作流

```
用户输入
  │
  ▼
┌─────────────────────────────────────┐
│ 步骤1：执行（Agent）                  │
│ - 加载 Skillbook                     │
│ - 渲染所有 active 策略为提示词片段     │
│ - 注入系统提示词                       │
│ - 执行任务，输出答案 + 推理轨迹        │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 步骤2：反思（Reflector）               │
│ - 分析本次执行轨迹                     │
│ - 识别：哪些策略有帮助？哪些有害？      │
│ - 提炼：有无新的可复用经验？            │
│ - 输出：策略评估标签 + 改进建议         │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 步骤3：更新（SkillManager）            │
│ - TAG：更新已用策略的计数               │
│ - ADD：新增有价值的新策略               │
│ - UPDATE：精炼已有策略                  │
│ - REMOVE：停用持续有害的策略            │
│ - 持久化到 JSON 文件                   │
└─────────────────────────────────────┘
```

| 3 | 更新时间 | 同分时按 updated_at 降序 |

改造实施清单

1. **阅读现有代码**：定位现有记忆文件路径和读写模块，理解当前注入逻辑
2. **实现 Skillbook 模块**：策略的 CRUD + 评分 + 序列化/反序列化（JSON）
3. **改造注入逻辑**：将"读取文本文件注入"改为"渲染 active 策略为 markdown 注入"
4. **实现 Reflector**：在 Agent 输出后，调用 LLM 分析本次交互，输出结构化反思结果
5. **实现 SkillManager**：根据反思结果执行 ADD/UPDATE/REMOVE/TAG 操作
6. **实现去重**：周期性调用嵌入模型，相似度超阈值则合并（可选，初期可跳过）
7. **数据迁移**：将现有文本记忆转换为 Skillbook 初始策略（每条文本 → 一条 Skill，active=true）
8. **改造预览 UI**：将纯文本预览框替换为卡片列表 + 编辑面板（详见第六章）
9. **保持接口兼容**：确保改造后现有调用方无感知

约束与注意事项

- 三步循环可共享同一个 LLM，无需额外配模型
- 反思步骤可异步执行，不阻塞主响应（若性能敏感）
- 去重需要嵌入模型（要求用户必须配置嵌入模型才可启用本全局记忆）
- 现有文本记忆需迁移为初始 Skillbook 数据，不能丢
- 预览 UI 的卡片方案是为了保证编辑零损耗，不要退回纯文本方案
- 用户手动编辑的策略应标记来源为 manual，避免被自动反思流程覆盖或删除
- 名称依然命名为全局记忆（前端显示下），但内核为ACE记忆机制
- 记忆系统默认使用全局下的默认模型，独立于agent框架自动运行（启用后），请避免提示词污染而导致的记忆准确性和精度
